# Как мы ведём репозиторий (GitHub Flow)

Методология — **GitHub Flow**: `main` всегда в рабочем состоянии, любая работа идёт
через короткоживущие ветки и Pull Request. Прямые пуши в `main` не делаем.

## Правила

1. **`main` — защищённый и всегда зелёный.** В него попадает только код, прошедший CI и ревью через PR.
2. **Одна фича = одна ветка.** Ветка маленькая и живёт недолго (часы–дни, не недели).
3. **Любое изменение `main` — через Pull Request.** Никаких прямых коммитов в `main`.
4. **CI обязателен.** PR можно мёржить только когда `CI` зелёный (см. `.github/workflows/ci.yml`).

## Цикл работы

```
main ──┐
       ├── feature/<имя>   ← создали ветку от свежего main
       │      commit, commit, push
       │      открыли PR → CI → ревью
       └──◄── merge в main  ← после зелёного CI
```

### По шагам

```bash
# 1. Свежий main
git checkout main
git pull

# 2. Ветка под фичу
git checkout -b feature/short-description

# 3. Работаем маленькими коммитами
git add -p
git commit -m "Понятное описание изменения"

# 4. Пушим ветку
git push -u origin feature/short-description

# 5. Открываем PR в main
gh pr create --base main --fill

# 6. Ждём зелёный CI, проходим ревью, мёржим
gh pr merge --squash --delete-branch
```

## Соглашения

- **Имена веток:** `feature/...`, `fix/...`, `docs/...`, `chore/...`.
- **Коммиты:** короткое императивное описание в первой строке («Add ...», «Fix ...»).
- **Размер PR:** чем меньше, тем быстрее ревью. Большую задачу режем на несколько PR.
- **Merge-стратегия:** `squash` — один коммит на фичу в истории `main`.
- **После мёржа** ветку удаляем (`--delete-branch`).

## Локальная проверка перед PR

```bash
pip install -r requirements.txt
pytest -q
```

Если тесты зелёные локально — CI с большой вероятностью тоже пройдёт.
